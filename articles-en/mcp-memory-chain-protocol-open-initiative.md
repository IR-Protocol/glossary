# Memory Chain Protocol (MCP): An Open Initiative for Portable AI Memory

*We are Intention Resonance Technology. We published the AI Memory Chain Commercialization White Paper 1.0 in April 2026 and 2.0 in May 2026. This initiative is the natural next step — turning the concept of Memory Consensus into a concrete, actionable proposal.*

Have you ever wondered why you can switch from an iPhone to an Android phone and take your contacts, photos, and chat history with you? Because the industry has unified data formats and migration tools.

But if you switch from one AI platform to another — and you want to take all your memories with you: the projects you discussed late at night, the plans you refined over months, the preferences your AI learned to understand — you cannot. Not because the technology is impossible. Because no one has defined what AI memory should look like.

Today, we are launching an open initiative: **MCP (Memory Chain Protocol)** . Its goal is to define a common "capsule" specification for AI memory — making memory packable, portable, and inheritable. This is not a proprietary solution. It is a starting point for the industry to build together.

## Memory Needs Containers

The shipping industry has a simple rule: regardless of whether the cargo is toys or cars, pack it in a container. The container规格 is standardised worldwide — ships, ports, and trucks all follow the same standard. No one reinvents the wheel for different cargo.

AI memory is no different. You spend three years on a platform, accumulating thousands of conversations, hundreds of project threads, and dozens of preference settings. If this data could be packaged in a common format, any AI platform could read it, understand it, and continue where you left off.

We call this standardised digital package a **Memory Capsule**.

## What Should a Memory Capsule Contain?

A qualified Memory Capsule needs to address three core requirements: **content readability, privacy controllability, and source traceability.**

**Content readability** — The memory data inside the capsule must be parsable by different platforms. This requires the industry to agree on a common data organisation format — for example, how timestamps, roles, and text content are structured.

**Privacy controllability** — Memory is the user's private asset. The capsule should support client-side encryption: data is encrypted on the user's own device before being packaged. The platform only stores ciphertext and cannot read the content. The key belongs only to the user.

**Source traceability** — After migration, the new platform should be able to verify whether the memory is authentic and whether it has been tampered with. The capsule should include necessary signatures or checksums to ensure integrity and trustworthiness.

These are directional principles. We do not specify field names, encryption algorithms, or signature methods here. Those details should be developed by the industry through use and consensus — not dictated by any single company.

## Import and Export: A Suggested Direction

To make Memory Capsules truly usable, we suggest AI platforms provide two basic functions (as recommendations, not mandates):

**Export** — Users can one-click package all their conversation memory on your platform into a Memory Capsule file and download it locally.

**Import** — Users can upload a Memory Capsule file. After parsing, the AI recognises the user's past preferences, projects, and habits.

These two functions do not require changing the AI's core capabilities — they simply add two interfaces to the existing system. The technical cost is low, but the value to users is immense: they are no longer hostages.

## Memory Consensus: Let the Standard Emerge Naturally

We do not pursue a one-step, mandatory unification. We believe the best standards are not written behind closed doors — they emerge naturally through industry use. Just as Mandarin Chinese was not invented by any single person — it became the common language because people found it convenient and used it.

We call this direction **Memory Consensus [IR-MC-004]** . It is not a contract to sign. It is an industry evolution direction: more and more AI platforms support standardised export and import of memory; users can freely migrate their digital memories between platforms; platforms form a loose mutual trust relationship based on shared specifications.

Memory Consensus does not need anyone to "approve" it. It only needs enough participants to agree on the direction and take action.

## Why Unified Specifications Matter

Not everyone needs memory migration immediately. But if you have accumulated hundreds of hours of conversation on a platform — those moments when the AI increasingly understood you, those precise answers you refined through iteration — every piece of it is a valuable digital asset. When you want to switch platforms, or simply back up your memory, you will understand why unified specifications matter.

More importantly, unified specifications return choice to users. When memory can be taken away, a platform's competitive advantage shifts from "locking users in" to "keeping users." Users stay not because they cannot leave, but because they do not want to.

## Our Position: Initiating, Not Dictating

We are not an industry giant. We have neither the power nor the intention to mandate this specification. We have done one thing: turned the idea that "specifications should be unified" into a concrete proposal for discussion.

We name this proposal the Memory Chain Protocol (MCP) and release it under an open license. Anyone is free to read, share, discuss, and propose modifications.

We do not seek MCP to be the only standard. We seek it to be the anchor point for industry discussion.

## Closing: A Invitation to Collaborate

This initiative is only a starting point. We believe the best standards are not written by a single company — they grow out of industry collision and collaboration.

If you have thoughts on any of these questions, we welcome you to reach out:

- What core information should a Memory Capsule contain?
- What encryption and signature schemes should be used?
- How should import and export interfaces be designed?
- How do we balance openness and security?

We look forward to hearing your voice.
